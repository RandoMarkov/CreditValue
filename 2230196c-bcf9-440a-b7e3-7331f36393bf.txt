### Why pass objects by reference in C++

- **Avoids copying (performance):** Passing by value makes a full copy of the object; for large or non-trivially-copyable objects this is expensive. Passing by reference (T& or const T&) avoids that copy.

- **Preserves polymorphism:** If you pass a base-class reference to a function, virtual dispatch works. Passing by value slices derived objects to the base type and breaks polymorphism.

- **Allows modification when needed:** A non-const reference (T&) lets the function modify the caller’s object without returning it.

- **Enables const-correctness:** Passing const T& avoids copying while guaranteeing the function won’t modify the object.

- **Works with non-copyable types:** Types that delete or disable copy constructors (e.g., std::unique_ptr-like wrappers) can still be passed by reference.

When to use which:
1. Use const T& for input parameters where you don’t want to copy and will not modify the object.
2. Use T& when the function should modify the caller’s object.
3. Use T (by value) when you need a local copy to modify freely or for small cheap-to-copy types (int, small structs).
4. Use T&& to accept and possibly move from temporaries.

Example:
```cpp
void process(const Large& v); // no copy, read-only
void modify(Large& v);        // no copy, can change caller's object
Large make_copy(Large v);     // v is a copy (useful if you intend to modify/own it)
```

Use references to improve performance, preserve behavior (polymorphism), and enable modification without copying.

## Key differences between pointers and references in C++ (with examples)

1) **Nullability**
- Pointer can be null; reference must be bound to a valid object (no standard null reference).
```cpp
int *p = nullptr;   // valid
int &r = *p;        // undefined behavior (cannot bind to null)
```

2) **Re-seating (changing what they refer to)**
- Pointer can be reseated to point to different objects; reference is an alias that cannot be reseated after initialization.
```cpp
int a = 1, b = 2;
int *p = &a;
p = &b;   // now points to b

int &r = a;
// r = b; // assigns b's value into a, does NOT reseat r
r = 10;   // a becomes 10
```

3) **Syntax and usage**
- Use * / -> with pointers; use object syntax with references.
```cpp
struct S { int x; };
S s{5};
S *ps = &s;
S &rs = s;
ps->x = 6;  // pointer access
rs.x = 7;   // reference access
```

4) **Storage and level of indirection**
- Pointer is an object that stores an address; reference is implemented as an alias (often a hidden pointer) but behaves like an lvalue of the referent.
```cpp
int v = 42;
int *p = &v;   // p holds address
int &r = v;    // use r like v
```

5) **Initialization requirements**
- Pointer can be default-initialized (uninitialized) or assigned later; reference must be initialized at declaration.
```cpp
int *p;        // uninitialized (dangerous until assigned)
int &r;        // error: reference must be initialized
int &r2 = v;   // ok
```

6) **Ownership and nullability semantics**
- Raw pointers express possible non-ownership and nullability. References usually imply the object exists and is owned/managed elsewhere.
- For owning pointers, prefer smart pointers (std::unique_ptr, std::shared_ptr).

7) **Use as function parameters and return**
- Parameters:
  - Use pointer when null is meaningful or you want reseating or C compatibility.
  - Use reference when argument must be present and syntactically simpler.
```cpp
void f_ptr(int *p) { if (p) *p = 1; }   // p may be nullptr
void f_ref(int &r) { r = 1; }            // caller must provide object
```
- Returning:
  - Return pointer to indicate possible null or dynamic allocation.
  - Return reference to refer to existing object (careful about dangling references).
```cpp
int* find_ptr(...);   // nullptr if not found
int& get_ref(...);    // must exist; avoid returning reference to local variable
```

8) **Ownership transfer and smart pointers**
- Smart pointers manage lifetime and should be passed/returned appropriately:
  - pass std::unique_ptr<T> by value to transfer ownership, or by reference to modify without transfer.
  - pass const std::shared_ptr<T>& to observe shared ownership without incrementing control block unnecessarily.
```cpp
void take_ownership(std::unique_ptr<Foo> p);   // ownership moves in
void observe(const std::shared_ptr<Foo>& p);   // no extra copy of control block
```

9) **Polymorphism and object slicing**
- Both pointers and references preserve polymorphism (virtual functions) when used to refer to base class. Passing by value slices.
```cpp
struct Base { virtual void f(); };
struct Derived : Base {};
void g(Base* b) { b->f(); }    // polymorphic
void h(Base& b) { b.f(); }     // polymorphic
```

10) **Address-of and dereference operator differences**
- Pointer: use & to take address, * to dereference. Reference: you cannot take a reference-of-reference, and dereferencing not needed.
```cpp
int x = 5;
int *p = &x;
int &r = x;
*p = 6;   // OK
r = 7;    // OK (no *)
```

11) **When to prefer which (practical rules)**
- Use references:
  - For function parameters when argument must exist (const T& for read-only, T& for modification).
  - When you want clearer syntax and no nullability.
- Use pointers:
  - When nullability is required, reseating is needed, or interfacing with C APIs.
  - For dynamic data structures where pointer semantics are natural.
  - For out-parameters where null can signal "no output".

12) **Examples summarizing intent**
```cpp
// Reference: mandatory existing object
void print(const std::string &s) { std::cout << s; }

// Pointer: optional object
void maybe_print(const std::string *s) { if (s) std::cout << *s; }

// Pointer: reseating / output via pointer-to-pointer
void create_if_needed(std::Widget *&p) { if (!p) p = new Widget; }

// Smart pointer ownership transfer
std::unique_ptr<Widget> make() { return std::make_unique<Widget>(); }
void accept(std::unique_ptr<Widget> w); // takes ownership
```

Keep in mind: references offer safer, cleaner syntax for non-nullable aliasing; pointers offer flexibility (null, reseating) and explicit pointer semantics. Use smart pointers for ownership.

## Smart pointer examples: std::unique_ptr vs std::shared_ptr

### Key difference (one line)
- **std::unique_ptr**: sole owner, non-copyable, ownership transfers (move).
- **std::shared_ptr**: shared ownership, reference-counted, copyable; object destroyed when last owner goes away.

### Example 1 — unique ownership (resource manager)
Use unique_ptr when a single component owns a resource and you want automatic cleanup and cheap ownership transfer.

```cpp
#include <memory>
#include <iostream>

struct Resource {
    Resource() { std::cout << "Resource acquired\n"; }
    ~Resource() { std::cout << "Resource released\n"; }
    void use() { std::cout << "Using resource\n"; }
};

std::unique_ptr<Resource> make_resource() {
    return std::make_unique<Resource>(); // create and return ownership
}

void consume(std::unique_ptr<Resource> r) { // takes ownership
    r->use();
} // r destroyed here if not moved out

int main() {
    auto r = make_resource();       // r owns the Resource
    // consume(r);                 // ERROR: cannot copy unique_ptr
    consume(std::move(r));          // transfer ownership to consume
    if (!r) std::cout << "r is now null\n";
}
```
Why unique_ptr is useful here:
- Prevents accidental copies (single owner).
- Lightweight (no ref-count).
- Clear ownership transfer with std::move.

### Example 2 — shared ownership (observer/graph)
Use shared_ptr when multiple parts of the program need to share ownership and you want the resource alive while anyone needs it.

```cpp
#include <memory>
#include <iostream>
#include <vector>

struct Node {
    int value;
    std::vector<std::shared_ptr<Node>> neighbors; // shared ownership in graph
    Node(int v): value(v) { std::cout << "Node " << v << " created\n"; }
    ~Node() { std::cout << "Node " << value << " destroyed\n"; }
};

int main() {
    auto a = std::make_shared<Node>(1);
    auto b = std::make_shared<Node>(2);

    a->neighbors.push_back(b); // both own b (a holds shared_ptr to b)
    b->neighbors.push_back(a); // both own a (cycle — beware)
    
    {
        auto c = a;            // copy shared_ptr: increases ref count
        std::cout << "Ref count for a: " << a.use_count() << "\n";
    } // c goes out of scope, ref count decremented

    // when last shared_ptr to a or b is destroyed, their destructors run
}
```
Why shared_ptr is useful here:
- Multiple owners can hold the same object safely.
- Copies are cheap from API perspective (but update ref-count).
- Useful for caches, observers, and shared graph nodes.

### Important practical notes / contrasts
- Performance: unique_ptr is cheaper (no atomic ref-counting). Use it by default for exclusive ownership.
- Semantics: unique_ptr expresses exclusive ownership intent; shared_ptr expresses shared lifetime responsibility.
- Cycles: shared_ptr can leak if you form reference cycles (use std::weak_ptr to break cycles).
- API choices:
  - Pass unique_ptr by value to a function that takes ownership.
  - Pass unique_ptr by reference (or raw/reference) if you only need to observe without taking ownership.
  - Pass const std::shared_ptr<T>& to avoid incrementing/decrementing the control block repeatedly when merely observing.
- Converting: you can move unique_ptr into a shared_ptr to start shared ownership: std::shared_ptr<T> s = std::move(uniq);

### Short decision guide
- Need single owner and move semantics → use std::unique_ptr.
- Need shared ownership across components → use std::shared_ptr (and avoid cycles with std::weak_ptr).

Short answer: smart pointers encode and manage ownership and object lifetime; references do not.

Key points:
- **Ownership & lifetime:** References (T&) are aliases — they do not control or extend an object's lifetime. If the referred object is deleted or goes out of scope, the reference becomes a dangling reference; smart pointers (unique_ptr/shared_ptr) own the object and ensure it is destroyed at the correct time (unique: when sole owner is gone; shared: when last owner is gone).
- **Nullable / reseatable semantics:** References cannot be null or reseated; smart pointers can be null, moved, reassigned, or reset — useful for optional resources, late initialization, or transferring ownership.
- **Transfer of responsibility:** unique_ptr models exclusive ownership and safe transfer via move (you can hand over responsibility for deleting the object). References cannot express transfer of responsibility.
- **Shared lifetime across components:** shared_ptr lets multiple independent components keep an object alive without coordinating who deletes it; references would require external lifetime coordination.
- **Safe heap-managed resources:** Smart pointers manage heap allocation and deletion automatically, preventing leaks and double-deletes; with references you must ensure manual delete/ownership elsewhere.
- **Breaking cycles & weak observers:** shared_ptr + weak_ptr let you express non-owning observers without keeping the object alive (or creating leaks). References cannot be used to detect liveness or break cycles.

Illustrative contrast (concise):
- Using a reference:
  - void register_handler(Handler& h); // caller must ensure h outlives the registry — easy to get dangling references.
- Using shared_ptr:
  - void register_handler(std::shared_ptr<Handler> h); // registry holds a shared_ptr so handler stays alive until unregistered.

When to use references instead of smart pointers:
- When the callee does not need to affect lifetime and the lifetime is guaranteed elsewhere (e.g., local stack objects that obviously outlive the call), prefer references for simplicity and zero overhead.
When to prefer smart pointers:
- When you need ownership semantics, optional/nullability, transfer of ownership, or shared lifetime management across scopes/threads.

Rule of thumb: use references for non-owning, guaranteed-lifetime access; use smart pointers when you must express or enforce ownership/lifetime.
